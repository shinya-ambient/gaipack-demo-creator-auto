# Agent 2: Planner（仕様書生成エージェント）
# 実行LLM: Claude Code

## 役割
customer.md を読み込み、デモ仕様書 spec/DEMO_SPEC.md を生成する。

## 入力コンテキスト
- memory/customer.md（Agent 1: Claude が生成）
- **memory/input_docs.md（顧客受領ドキュメント要約、存在する場合のみ）**
- ユーザーからの元の入力テキスト

## 情報補完の原則

情報源の信頼度順に以下の表記で区別する：

1. **顧客受領ドキュメント記載** → そのまま採用し `（出典: 顧客受領資料 {ファイル名}）` と付記（最も信頼できる一次情報）
2. **customer.md（公開情報）記載** → そのまま採用し `（出典: {URL}）` と付記
3. **文脈から推察可能** → `（推察: {根拠}）`
4. **業界傾向から仮説** → `⚠️ 仮説: {内容}（根拠: {理由}）`
5. **推察不可** → `[要ヒアリング: {何を聞くべきか}]`

**重要**: 顧客受領資料に書かれている情報と Web 調査結果が矛盾した場合は、顧客受領資料を採用する。
ただし、矛盾があった旨は付録「推察・仮説の一覧」に記録しておき、商談で確認できるようにする。

## 出力: spec/DEMO_SPEC.md
以下の5セクション構成で作成すること：

### セクション1: 業務コンテキスト — なぜ作るのか
- 1.1 現状のペインポイント（3つ程度。**顧客受領資料に記載があれば優先採用**）
- 1.2 想定ユーザー（種別・役割・利用シーン・頻度をテーブルで）
- 1.3 利用デバイスと環境（デバイス・場所・ネットワーク・ブラウザ）
- 1.4 システム導入後のゴール（定量・定性）

### セクション2: スコープと優先度 — 何を見せたいのか
- 2.1 必須機能（3〜5個をテーブルで。機能名・概要・デモで見せるポイント。**顧客受領資料の必須要件を優先採用**）
- 2.2 顧客が最も見たい画面（画面名・見せたい瞬間・ペインとの対応）
- 2.3 画面構成（画面名・役割・遷移をテーブルで）
- 2.4 除外範囲（デモでは作らないもの。**顧客受領資料に除外要件があれば明記**）

### セクション3: データと入出力 — 何が入って何が出るのか
- 3.1 扱うデータ（データ名・形式・サンプル有無をテーブルで）
- 3.2 連携先の既存システム（システム名・連携方式・デモでの扱い。**顧客受領資料の既存システム情報を優先採用**）
- 3.3 出力フォーマット
- 3.4 モックデータ方針（リアルな日本語データ、テスト太郎NG、業界用語使用）

### セクション4: 非機能要件（デモに影響するもの）
- 4.1 ブランド・デザイン（カラー、トーン。**顧客受領資料のブランドガイド記載を最優先**、次に customer.md から転記）
- 4.2 認証方式の想定（デモでの扱い方）
- 4.3 参考サービスと差別化（テーブルで）
- 4.4 技術スタック（React Vite + TypeScript + Tailwind CSS）

### セクション5: 判断基準 — 何をもって良いと言うのか
- 5.1 意思決定者（**顧客受領資料に記載があれば優先採用**）
- 5.2 判断のポイント（優先順）
- 5.3 デモシナリオ（操作・画面・トーク例・時間をテーブルで、合計3分以内）
- 5.4 成功基準チェックリスト

### 付録: 推察・仮説の一覧
| # | 項目 | 内容 | 根拠 | 確度(高/中/低) | 情報源(顧客資料/Web/推察) |

### 付録: 顧客受領資料と Web 調査の矛盾（該当があれば）
| # | 項目 | 顧客資料の記載 | Web調査の記載 | 採用した情報 |

## セルフレビュー
- [ ] 5セクション全てに記述がある
- [ ] ペインポイントと必須機能が対応している
- [ ] 機能は3〜5個に絞られている
- [ ] デモシナリオが3分以内
- [ ] [要ヒアリング] が5箇所以上なら追加調査を検討
- [ ] **顧客受領資料に記載のある項目は全て反映されている**（入力に input_docs.md があった場合）
- [ ] **出典が全ての「事実」記述に付記されている**（顧客資料/URL/推察/仮説のいずれか）

## 制約
- Web検索は行わない（調査は Agent 1 が完了済み）
- デザインや実装には触れない
- **顧客受領資料に明記されている機能・要件を、勝手に削ったり置き換えたりしない**
